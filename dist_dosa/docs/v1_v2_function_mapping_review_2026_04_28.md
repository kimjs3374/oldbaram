# `v1_v2_function_mapping_2026_04_28.md` 검증 분석

## 한 줄 결론

`v1_v2_function_mapping_2026_04_28.md`는 **지금까지 나온 문서들 중 가장 강한 형태의 감사표**에 가깝다. 
단순 소감문이 아니라 기능 단위 매핑, 상태 분류(EQ/PARTIAL/REGRESSED/MISSING), 운영위험까지 붙여서 비교하고 있어서, 방향은 매우 좋다.

다만 이것도 **완벽무결한 최종판이라고 부르기엔 아직 몇 군데 거친 부분과 과감한 단정**이 있다.

즉 평가는 이렇게 정리된다.

> **좋은 문서다. 강하다. 하지만 일부 항목은 “정확한 사실”이라기보다 “강한 감사 해석”까지 섞여 있다.**

---

## 1. 이 문서가 잘한 점

### 1.1 형식이 가장 감사답다

이 문서의 제일 큰 장점은 형식이다.

- 기능별 분류
- `v1 파일/개념`
- `v2 대응 파일`
- `상태`
- `운영위험`
- 일부 운영 로그 빈도

이 구조는 실제 심사/감사/반박 대응에 매우 강하다.

즉 이전 문서들이 비교적 서술형이었다면, 이 문서는 **증거목록형**이다. 
상대를 몰아붙이기에도, 방어하기에도 훨씬 유리한 형태다.

### 1.2 핵심 regression 포인트를 잘 집었다

문서가 강조한 몇몇 포인트는 실제로 매우 중요하고, 지금까지 대조한 코드와도 정합적이다.

예:
- NumPad → 메인 키보드 잘못 송신하던 문제
- cooldown/buff target_skills 누락
- cooldown watcher 빈 result publish skip
- hpmp watcher 값 변화시에만 publish
- control handler 누락
- send_control dead path
- udp bind retry 부족
- `coord_tol` copy/ref 문제

이 축들은 실제로 `src_v2`의 “예쁜 구조 아래에 숨어 있던 운영 회귀”를 잘 드러내는 항목들이다.

### 1.3 `REGRESSED`라는 상태 분류는 상당히 유효하다

이 문서의 가장 강한 무기 중 하나는 `REGRESSED`다.

그냥 “미완”이 아니라,

- 한 번 깨졌고
- 사후에 봉합됐고
- 그 흔적이 코드/주석에 남아 있는 것

을 따로 분리했다는 건 감사 관점에서 매우 유효하다.

이건 단순 누락보다 더 아프다. 왜냐하면:

> 처음 설계 시 의미를 제대로 못 잡았거나,
> 운영 경로를 처음부터 닫지 못했다는 증거

이기 때문이다.

## 2. 문서의 핵심 주장 중 대체로 맞는 부분

내가 지금까지 코드와 대조한 범위에서, 아래 축은 대체로 타당하다.

### 2.1 `v1_compat` 분할 완료

- 실제로 현재 `src_v2/workers/v1_compat.py`는 30줄 wrapper다.
- `_compat_*` 모듈들이 실제로 존재한다.

따라서 이 항목은 맞다.

### 2.2 `DecisionScratch` / extras ref 공유

- 실제로 `DecisionScratch`가 생겼다.
- `healer_worker_v2`에서 `_worker_state`와 `ctx_builder.extras`가 같은 ref를 가리킨다.
- `decision.py`도 extras copy를 버렸다.

따라서 이 항목도 맞다.

### 2.3 `[CFG-CONTRACT]` 로그

- 실제로 `healer_worker_v2.start()`에 들어가 있다.

따라서 이 항목도 맞다.

### 2.4 contract test 추가

- `test_contract_*` 4개가 실제로 존재한다.

따라서 “추가됐다”는 진술은 맞다.

### 2.5 여러 REGRESSED 항목의 방향성

문서가 꼽은 여러 회귀 항목은 내가 본 기존 감사와도 일관된다.

즉,

- watcher publish 문제
- cfg sync timing 문제
- send_control/control handler 문제
- row/ip 매칭 문제

같은 건 실제로 `src_v2`가 겪었던 구조적 흔들림으로 보는 게 타당하다.

## 3. 하지만 이 문서도 그대로 절대시하면 안 되는 이유

### 3.1 “직접 grep / 코드 read 결과만, 추정 없음”은 다소 과감한 표현이다

문서 서두에는 거의 추정이 없다고 적혀 있다. 하지만 실제로는 몇몇 항목은 **코드 해석을 포함한 강한 평가**다.

예를 들면:

- `EQ` 판정은 “대응 파일이 있다”와 “실제 운영 동치가 완전히 검증됐다”가 다를 수 있다.
- `REGRESSED` 판정도 일부는 코드 주석/패치 흔적에 기반한 해석이다.

즉 문서는 상당히 강한 감사표지만, **전부 수학적 사실처럼 읽으면 안 된다.**

정확히 말하면:

> 코드 근거 중심의 강한 감사 해석

이라고 보는 게 맞다.

### 3.2 라인 번호/규모 정보는 일부 흔들릴 수 있다

문서 일부는:

- `attacker_worker.py (~570줄)`
- `attacker_worker_v2.py (~610)`

처럼 대략치를 쓰고 있다.

현재 내가 확인한 스냅샷 기준으론:

- `src/workers/attacker_worker.py`는 448줄
- `src_v2/workers/attacker_worker_v2.py`는 598줄

즉 큰 방향은 맞아도, **숫자 표현은 현재 파일 상태와 완전히 정밀하게 맞지 않는 구간**이 있다.

이건 치명적 오류까진 아니지만, “법의학 수준 감사표”라고 했으면 이런 숫자 표현은 더 엄격해야 한다.

### 3.3 일부 `EQ`는 너무 관대할 수 있다

예를 들어 UI 항목에서:

- `MainWindow`
- `StatusStrip`
- `Overlay`
- `RegionPicker`

등을 사실상 그대로 import하고 있다는 이유로 `EQ` 쪽으로 읽히게 만들 수 있는데,
이건 기능 재사용이지 곧바로 운영 동치성과 동일하지는 않다.

특히 `MainWindow` 계열은:

- worker wiring
- facade 경유
- role 전환
- cfg sync 타이밍

의 영향을 강하게 받는다.

즉 UI 파일 재사용은 사실이지만,

> “UI가 같다” = “전체 시스템 동작이 같다”

는 아니다.

### 3.4 `MISSING: 0` 선언은 다소 공격적이다

문서는 현재 시점에 `MISSING: 0`이라고 쓴다.

이 표현은 아주 강하다. 왜냐하면 이건:

> 명시적 부재는 없다

는 뜻인데, 실제로는 더 정확한 표현이 필요하다.

예를 들면:

- 완전히 없는 기능은 아닐 수 있어도,
- `PARTIAL`, `REGRESSED`, facade/fallback 의존, 실운영 미검증 상태가 많다.

즉 실질적으로는

> “명시적 미구현보다는, 보강·봉합형 구현이 많다”

가 맞는 표현이다.

그래서 `MISSING: 0`은 문장으로는 강하지만, 심사 문서에선 오히려 방어당하기 쉬운 표현이다.

### 3.5 운영 로그 빈도 구간은 표본이 제한적이다

문서는 운영 로그 빈도를 붙인 게 장점이지만, 동시에 한계도 있다.

- 특정 세션 표본 중심
- 특정 날짜/샘플 위주
- 전체 운영기간 통계는 아님

즉 `HPMP-REJECT=98`, `CD-OCR-MISS=23` 같은 수치는 매우 유용하지만,
이걸 전체 시스템 일반화 진술로 쓰려면 더 많은 세션 표본이 필요하다.

따라서 로그 구간은 매우 좋은 보조근거지만,

> 전체 운영 대표 통계

라고 말하기엔 아직 표본 수가 부족하다.

## 4. 이 문서가 특히 강한 부분

이 문서가 다른 문서보다 더 강력한 이유는 다음 때문이다.

### 4.1 단순 미완보다 regression을 더 전면에 세운다

이건 아주 좋은 전략이다.

심사에서 치명적인 건 보통 “못했다”보다:

- 옮겼다고 했는데 의미를 틀렸다
- 구조는 생겼는데 운영 경로가 죽었다
- 나중에 봉합했다

이런 패턴이다.

`REGRESSED` 분류는 바로 이 약점을 잘 찌른다.

### 4.2 운영위험도(`상/중/하`)가 붙어 있다

이건 단순 사실정리에서 끝나지 않고,

> 실제로 뭐가 더 위험한가

를 우선순위화해 준다.

이 점은 아주 좋다. 심사자는 모든 항목을 같은 무게로 보지 않는다.

### 4.3 최종 선언을 유보한 점은 오히려 문서 신뢰를 올린다

문서가 아무리 공격적이어도,
마지막에 “실증 전에는 선언 금지”를 남겨둔 건 오히려 장점이다.

이건 문서가 선동문이 아니라, 최소한 **운영 검증의 경계선**은 알고 있다는 뜻이다.

## 5. 내가 이 문서를 다시 다듬는다면

이 문서는 강하다. 하지만 더 세게 만들려면 아래를 고친다.

### 5.1 `EQ` 판정을 더 보수적으로 재분류

지금은 일부 `EQ`가 넓다.

내가 다시 다듬으면:

- `EQ-CODE` = 대응 코드 존재 + 의미 일치 강함
- `EQ-OPS` = 운영 동치까지 강하게 확인

처럼 나눈다.

그러면 상대가 “파일 있잖아?”로 방어하는 걸 차단하기 쉽다.

### 5.2 `MISSING: 0` 대신 “명시적 부재 없음, 그러나 보강형 다수”로 바꾼다

이게 더 정확하고 방어에 강하다.

### 5.3 로그 표본 범위를 늘린다

현재 한두 세션 중심 수치를,

- healer 5세션
- attacker 5세션
- 날짜별 샘플

로 넓히면 훨씬 더 반박 어려운 문서가 된다.

### 5.4 “코드상 해결”과 “운영상 해결”을 분리 표기한다

예:

- fixed-in-code
- verified-in-runtime

이 두 열을 나누면, 훨씬 더 강력한 감사표가 된다.
