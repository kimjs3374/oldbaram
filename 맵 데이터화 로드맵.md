# 맵 데이터화 로드맵 (2026-06-12 설계 합의)

> 목표: 옛바의 **각 맵을 좌표 그리드로 통째 데이터화**하고, 그 데이터 위에서
> 경로 최적화 · 장애물 자동 회피 · (최종) 힐러 단독 자율 사냥을 파생시킨다.
> 격수·힐러의 이동은 데이터를 **수집하는 수단**일 뿐이고, 진짜 산출물은
> **맵 데이터(grid) 그 자체**다.
>
> 이 문서는 미래 설계/로드맵이다. 현재 동작 명세는 `맵이동 로직.md` 참조.

## 0. 왜 이 방식인가 (제약이 설계를 결정한다)

옛바는 다음 제약을 가진다 (메모리 `project_nexon_anticheat`, `project_no_minimap`):

- **메모리 접근 불가** (넥슨 안티치트). 좌표/지형을 메모리에서 못 읽는다.
- **미니맵 없음.** 맵 전체 구조를 한눈에 주는 UI가 없다.
- **화면은 캐릭터 중심 스크롤뷰.** 화면 픽셀로 지형을 읽어도 절대좌표로
  변환할 기준점이 없다 (그래서 중앙 prior·픽셀 임계·템플릿 전부 폐기됨,
  메모리 `feedback_forbidden_approaches`).

→ **맵을 데이터화할 입력은 단 하나: 우하단 좌표 OCR (절대 x,y 정수).**

이 단일 입력의 성격이 좋은 점:
- 좌표가 **정수** (`raw 0019xxxx` = x,y 각 4자리) → 그리드 셀로 바로 쓴다.
- 이동이 **4방향 직접** (사다리/점프 없음, 메모리 `project_movement_model`)
  → occupancy grid + A* 격자탐색이 **천생연분**. 플랫포머였다면 불가능.

→ 결론: 맵 데이터의 모든 셀 정보는 **"누가 그 좌표에 있었나"의 누적 관측**으로
채운다. 학습은 알파고식 RL이 아니라 **방문 카운트 누적**이 한다.
(RL은 시뮬레이터+리워드 루프가 필요한데, 실시간 게임 한 판이 곧 환경이라
학습 사이클이 느리고 밴 위험이 크다. 누적 통계는 같은 효과를 안전하게 낸다.)

## 1. 데이터 모델 — `maps/<맵명>.json` (맵 1개 = 파일 1개)

> **셀 크기 = 좌표 1단위 = 1셀** (실측 2026-06-12: 선비족입구 x∈1~4·y∈9~11,
> 선비족4 x∈22~26·y∈9~13. 맵당 수십 타일 규모의 작은 정수라 묶을 필요 없음).

```jsonc
{
  "map": "선비족4",                            // canonical 맵명 (서브맵 (N) 구분 유지)
  "bounds": { "x": [min, max], "y": [min, max] }, // 관측 좌표 범위 = 맵 크기
  "cells": {                                  // 희소 그리드. "x,y" → 셀 속성
    "22,9": { "walk": 312, "tab": 18 }        // walk=방문 누적수, tab=빨탭 관측수
    // walk 높을수록 검증된 통행로. tab 높을수록 사냥 명당.
  },
  "blocked": {                                // ★ 간선 차단 — STUCK 로그 직접 추출
    "4,11": { "L": 6 }                        // "이 좌표에서 L 방향은 벽" (관측 횟수)
    // A*에서 해당 (좌표,방향) 간선을 차단/고비용 처리. 장애물의 1차 증거.
  },
  "portals": [                                // portals_v2.json 흡수/통합
    { "at": [x, y], "exit_dir": "R", "to": "선비족5" }
  ],
  "sessions": 27,                             // 누적 학습 세션 수 (많을수록 신뢰)
  "updated": "2026-06-12T..."
}
```

### 핵심 전환: "선(trail)" → "셀(occupancy) + 간선(blocked)"

현재 `_map_trail`은 deque(선)이라 휘발·세션 단발이다. 이걸 **셀 카운트 누적 +
간선 차단 누적**으로 바꾸면:
- 격수가 그 맵을 100번 돌면 통행로 셀이 진해진다 (walk↑).
- `[STUCK] blocked` 가 쌓이면 "어느 좌표의 어느 방향이 벽"이 직접 드러난다.
- 세션을 넘어 영구 누적되어 데이터가 굳는다.

### 장애물은 추론이 아니라 STUCK 로그가 직접 준다 (사용자 통찰, 실측 검증)

힐러 로그 `[STUCK] h=(4,11) blocked=L ortho1=U ortho2=D` 한 줄이 곧
**"좌표 (4,11)에서 L 이동은 막힘"** 이라는 간선 차단 사실이다. "주변은 밟혔는데
자신은 never-walk" 같은 셀 추론보다 훨씬 직접적·정확하다. never-walk 추론은
STUCK 데이터가 닿지 못한 구역의 **보조**로만 쓴다.

### 데이터 소스 매핑 — 전부 **이미 쌓인 로그**에서 추출 (재수집 불필요)

| 셀/필드 | 로그 소스 (실측 라인) | 비고 |
|---|---|---|
| `walk` | `[attacker] coord=(x,y) valid=1` + 힐러 `[TRAIL-PUSH] coord=(x,y)` | 둘 다 walkable 증거. 누적 카운트 |
| `tab` | `[attacker] coord=(x,y) red_tab=1` | 빨탭 뜬 좌표 = 사냥 스팟 밀도 |
| `blocked` | 힐러 `[STUCK] h=(x,y) blocked=DIR` | ★ 막힌 좌표+방향. 장애물 1차 증거 |
| `portals` | 기존 `portals_v2.json` | 이미 영구 학습 중. 흡수만 |
| `bounds` | walk 셀들의 min/max | 맵 경계 |

## 2. 단계 로드맵 (순서 강제 — 데이터가 상위의 토대)

### S1. 로그 백필 파서 (Backfill) — **위험 0, 최우선, 코드 변화 무**
- **게임/앱을 안 건드린다.** 이미 쌓인 `logs/` + `logs_cloud/` 로그 파일을
  파싱해 `maps/<맵명>.json` 을 생성. 오프라인 분석 스크립트 1개.
- 추출: `[attacker] coord valid=1 red_tab` → walk/tab, `[TRAIL-PUSH]` → walk,
  `[STUCK] h blocked` → blocked 간선. (S0 실시간 수집기는 데이터가 검증된
  뒤에 붙여도 됨 — 백필이 먼저 가치를 증명한다.)
- 산출물: 수백 개 기존 세션 로그가 즉시 맵 데이터로 환생.

### S2. 경로 최적화 (Optimizer) — **사용자의 핵심 목표**
- 같은 맵의 누적 셀에서 입구→포탈(또는 스팟) **최단·고신뢰 경로**를 뽑는다
  (walk 진한 셀 = 검증된 간선, blocked 간선 = 회피). 자동사냥은 이 최적
  경로 위를 이동하며 전투한다.
- `hunt_analytics` 시간/경험치 연계로 "빠르고 끊김 없던" 경로 가중.

### S3. A* 경로탐색 + 장애물 회피
- `cells`(walkable) + `blocked`(차단 간선)로 격자 그래프 구성 → A* 로 임의
  두 좌표 간 경로 생성 → 장애물 자동 우회. 4방향 이동이라 격자 A*가 그대로 맞음.
- blocked 데이터가 닿지 않은 구역은 never-walk 셀 추론으로 보조.
- **→ 2026-07-05 NavBrain 으로 실현** (`경로 딥러닝 학습.md` 정본):
  flow field(Dijkstra, walk/blocked/막힘률 비용) + 격수 발자국 behavior
  cloning ONNX + 기권 게이트 + shadow 선검증. v72/v113 A* 롤백 교훈
  (데이터 희소 시 오확신)을 기권 게이트로 구조화.

### S4. 힐러 단독 자율 사냥 (최종)
- 격수 의존 제거. 힐러가 `tab` 높은 스팟으로 A* 이동 → 빨탭 자가 탐색 →
  전투 → 포탈로 맵 그래프 순회.
- 가장 크고 위험한 단계. S1~S3 데이터·검증이 선행돼야 함.

## 3. 한 데이터가 비전 4개를 전부 떠받친다

| 비전 | 사용하는 데이터 |
|---|---|
| 장애물 회피 | `obstacles` (walk=0 안쪽 셀) → A* 우회 |
| 최적 경로 | walk 진한 셀 = 검증된 간선 → 입구→포탈 최단 |
| 자율 사냥 | `tab` 높은 셀 = 명당 → 거기로 이동 후 전투 |
| 맵 전체 네비 | `portals` = 맵끼리 그래프 연결 |

## 6. S0 실시간 수집 + 클라우드 — 구현 설계 (2026-06-13)

> S1·S2 프로토타입 검증 완료(아래 §7). 이제 데이터를 **앞으로 자동 누적**해
> 계속 고도화한다. controller 핫패스를 건드리므로 신중히.

### 6.1 공용 모듈 `MapGrid` (구현 완료) — 위치 `src/fsm/map_grid.py`
- **괄호보정(`canon_map_name`)은 백필·런타임 단일 정본**: `_backfill_maps.py`
  가 import 해서 사용(중복 제거 완료). 누적/직렬화부는 백필이 sessions 집계를
  위해 독자 유지 → 완전 통합은 후속(반성문 #15: 합치기 항목 기록).
- 메모리 상태: `cells{"x,y":{walk,tab}}`, `blocked{"x,y":{dir:cnt}}`, dirty 플래그.
- API(전부 O(1), 디스크 IO 없음):
  - `add_walk(map, x, y)` / `add_tab(map, x, y)` / `add_blocked(map, x, y, dir)`
  - `flush()` — dirty 맵만 `maps/<맵>.json` 에 **머지 저장**(기존 카운트 + 신규).
  - `load_merge(other)` — 클라우드/타 PC 파일 카운트 합산(교환법칙 성립=충돌 없음).

### 6.2 런타임 훅 (controller, 힐러 PC 단일 수집)
UDP State에 `red_tab`이 실려 오므로 **격수·힐러 데이터를 controller 한 곳**에서.
- **격수 walk/tab**: `controller.py:672 trail.append(coord)` 직후 →
  `grid.add_walk(s.map_name, *coord)`, `s.red_tab` 면 `add_tab`.
- **힐러 walk**: `note_healer_map(healer_coord)` 유효 시 → `add_walk(healer_map, ...)`.
- **blocked**: `_decide_move` STUCK-ORTHO 가 blocked 방향 정하는 지점 →
  `add_blocked(healer_map, hx, hy, dir)`.
- **flush**: 핫패스에 디스크 IO 금지. 30초 throttle + 앱 종료 hook 에서만 flush.

### 6.3 클라우드 동기화 (`cloud_sync` 재사용, [[project_cloud_system]])
- 앱 시작: 클라우드 `maps/` pull → `load_merge` (다른 힐러 데이터 흡수).
- 주기/종료: 자기 `maps/` push. 머지가 카운트 합산이라 여러 힐러(37,120)가
  동시에 쌓아도 안전. 상대 빈도 보존 → A* 영향 무결.

### 6.5 격수(인간) 기반 막힘률 — 몹 vs 진짜 벽 구분 (사용자 지시 2026-06-13)

> 격수는 사람이 눈으로 보고 직접 조작 = 가장 정확한 데이터. 장애물을 힐러
> STUCK(봇 삽질)에 의존한 건 본말전도. **격수 walk = golden, occupancy 1차**,
> 힐러 STUCK = 보조. 추가로 격수의 "눌렀는데 못 감"을 직접 잡는다.

**핵심**: 같은 좌표라도 **몹 때문에 일시적으로 막힐 수 있다.** 단발 판정 금지.
누적 통계 = "시도 대비 막힘률"로 구분:
- 격수가 (x,y)에서 D키 누름 → 다음 프레임 좌표 관찰.
  - 좌표가 D로 변함 → `try+1` (그 방향 통행 성공).
  - N프레임(≈0.5s) 안 변함 → `try+1, block+1` (막힘 = 몹 or 벽).
- **막힘률 = block / try**. 항상 막힘(≈1.0)=진짜 벽, 가끔(낮음)=몹/일시.

**데이터 모델 확장**: `blocked "x,y" → {DIR: {try, block}}` (기존 cnt → try/block).
A* 간선 비용 = 막힘률 가중(높을수록 회피, 임계 초과는 차단). 계속 수집·학습해
보완(데이터 쌓일수록 몹/벽 구분 선명).

**구현 경로**:
- `skill_range_overlay` 의 방향키 감지(Win32 ↑↓←→ + msw foreground)를 공용
  함수로 추출 → attacker 도 사용.
- `protocol.State` 에 `atk_key` 필드(격수 실제 키방향) 추가. attacker 송신.
- controller: `atk_key` + 좌표 델타로 try/block 누적(`grid.add_attempt`).

### 6.4 배포
- `dist_dosa` 정본 수정 → `src` sync → install.bat 으로 양 PC([[project_cloud_system]]).
- 1차는 **로컬 누적만**(6.1~6.2) 검증 후, 클라우드(6.3) 붙인다. 단계 분리.

  수정으로 마감 (반성문 #10), 스모크 테스트 후 배포.
- 코드 수정 시 이 문서와 코드를 **함께 업데이트** (CLAUDE.md 확인 원칙).
